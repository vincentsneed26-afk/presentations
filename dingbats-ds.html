Publish-ready · team review edition

Dingbats, designed with calm precision.

A complete design system for product, marketing, and editorial surfaces. It balances warmth and utility, supports scale, and gives the Dingbats team one shared language for decisions, implementation, and critique.

What this review covers

  • Foundational decisions: color, typography, grid, spacing, motion, and accessibility.
  • Forty core UI components with anatomy, states, usage guidance, and implementation cues.
  • Patterns, token strategy, and engineering guidance for a durable design-to-code workflow.
Prepared for team review, alignment, and iteration.
Principles

System values

These principles define how the interface should feel and how trade-offs should be resolved when the system is extended.

Clarity first

Hierarchy, spacing, and behavior should explain the interface before decoration or explanation is needed.

Calm precision

The product should feel intelligent and composed. Motion is brief, emphasis is controlled, and noise is removed.

Accessible by default

Contrast, labels, focus, touch targets, and keyboard support are default requirements, not finishing tasks.

Foundations

The structural layer

The system is neutral-led, editorial in tone, and operationally sharp. Primary red orients action, while surfaces and type do most of the communicative work.

Brand posture

Playful in name, not childish in expression. Warm, literate, composed, and quietly witty.

Visual bias

Neutral surfaces, restrained color, subtle editorial softness, and disciplined interface geometry.

Product bias

Reduce noise, support reading and making, and preserve one clear action per region whenever possible.

Color

Neutral-led with a bright red primary

Use color to orient action and feedback, not to decorate structure. Most of the interface should live in neutral surfaces.

Usage rules

  • Use neutral surfaces for 80–90% of interface area.
  • Use one dominant primary CTA per view.
  • Prefer tinted surfaces over thick status borders.
  • Do not use color as the only state indicator.

Layout

12-column grid, 8px rhythm

The system aligns to a 12-column desktop grid, 8-column tablet grid, and 4-column mobile grid. One intentional break per page is enough.

Grid rules

  • Desktop: 12 columns, 24px gutter, 32px outer margin.
  • Tablet: 8 columns, 20px gutter, 24px outer margin.
  • Mobile: 4 columns, 16px gutter, 16px outer margin.
  • Forms generally cap at 6 columns for readability.
Spacing

An 8px system with disciplined exceptions

Motion

Brief, useful, and calm

Controls should react quickly. Containers can take slightly longer. Motion should communicate state change, spatial relationship, and confirmation, never spectacle.

Fast

160ms for controls, hover, pressed, and small state changes.

Standard

240ms for menus, drawers, toasts, and local view transitions.

Large

320ms for modal entry, layout shifts, and broader context changes.

Components

Forty core building blocks

Each component follows the same review model: anatomy, states, usage guidance, accessibility requirements, and code cues.

Primary, secondary, tertiary, destructive
  • AnatomyContainer, label, optional leading/trailing icon, focus ring.
  • UsageOne primary action per region.
  • Accessibility44px minimum height, loading preserves context.
  • Code44/52px heights, padded via spacing tokens.

2. Icon button

Utility and toolbar action
  • AnatomyFrame, icon, focus ring, tooltip trigger area.
  • UsageToolbar actions, dismiss, utilities.
  • AccessibilityRequires clear `aria-label`.
  • Code40px min, 44px on touch surfaces.

3. Segmented control

View switching for related options
  • AnatomyTrack, segments, active indicator.
  • UsageSwap tightly related views.
  • AccessibilityTablist or radio semantics depending on behavior.
  • CodeIndicator moves with transform.

4. Link

Navigation, not button behavior
  • AnatomyLabel, optional icon, underline cue.
  • UsageNavigation and references.
  • AccessibilityHover/focus must not rely on color only.
  • CodeUnderline offset 2px.

5. Text field

Short text input
  • AnatomyLabel, field, helper, error, prefix/suffix.
  • UsageName, title, URL, compact content entry.
  • AccessibilityAssociate help and error through IDs.
  • Code48px height, 12px × 16px padding.

6. Text area

Longer freeform input
  • AnatomyLabel, field, count, helper or error.
  • UsageDescriptions, prompts, notes.
  • AccessibilityConvey limit and validation text programmatically.
  • CodeMin-height 120px, vertical resize only.

7. Search field

Search and command entry
  • AnatomySearch icon, input, clear action.
  • UsageTop-level search and large data surfaces.
  • AccessibilityClear action needs explicit label.
  • CodeSupports debounce and keyboard results.

8. Select

Constrained option list
  • AnatomyLabel, value, chevron, menu.
  • UsageFinite options, compact forms.
  • AccessibilityPrefer native select on mobile where possible.
  • CodeMenu max-height 320px.

9. Combobox

Searchable structured selection
  • AnatomyInput, current value, result list.
  • UsageTags, assignees, long option sets.
  • AccessibilityFollow ARIA combobox patterns accurately.
  • CodeVirtualize lists over 200 items.

10. Checkbox

Multiple independent selections
  • AnatomyBox, mark, label, help.
  • UsageIndependent choices and multi-select lists.
  • Accessibility44px hit target, indeterminate supported.
  • CodeTree selection can expose partial state.

11. Radio

Single visible choice
  • AnatomyControl, dot, label, helper.
  • UsageMutually exclusive options.
  • AccessibilityGroup inside fieldset and legend.
  • CodeUse card variants only when description length requires it.

12. Switch

Immediate binary setting
  • AnatomyTrack, thumb, optional state label.
  • UsageOn/off settings with immediate effect.
  • AccessibilityLabel should describe result.
  • CodeRecommended size 52×32px.

13. Slider

Range adjustment
  • AnatomyTrack, fill, thumb, value cue.
  • UsageImmediate value tuning.
  • AccessibilityArrow-key support required.
  • Code24px minimum thumb target.

14. Stepper

Bounded numeric control
  • AnatomyMinus, value, plus.
  • UsageQuantities and bounded increments.
  • AccessibilityButtons need explicit action labels.
  • CodeHandle long-press carefully.

15. Date picker

Date and range selection
  • AnatomyField trigger, calendar, navigation, day cells.
  • UsageScheduling and filtering.
  • AccessibilityFull keyboard calendar behavior required.
  • CodeMobile should prefer sheet-style presentation.

16. Tag / chip

Filter and selected value token
  • AnatomyContainer, label, icon/avatar, remove affordance.
  • UsageFilters, metadata, selected values.
  • AccessibilitySeparate removal target must be focusable.
  • CodeOffer compact and default sizes.

17. Badge

Short status marker
  • AnatomyPill, short label, optional dot.
  • UsageStatus, counts, labels.
  • AccessibilityMaintain high contrast in tinted fills.
  • Code24px minimum height.

18. Tooltip

Supplementary clarification
  • AnatomyTrigger, floating label, optional arrow.
  • UsageClarify icons and dense controls.
  • AccessibilityReveal on focus and dismiss with escape.
  • Code240px max width.

19. Popover

Anchored contextual surface
  • AnatomyTrigger, container, body, optional actions.
  • UsageContextual content and lightweight tasks.
  • AccessibilityManage focus intentionally on open/close.
  • Code8px anchor offset.

20. Dropdown menu

Secondary action list
  • AnatomyTrigger, menu, items, shortcuts, dividers.
  • UsageUtility actions and compact menus.
  • AccessibilityArrow navigation and escape close.
  • Code36–40px item height.

21. Command palette

Power search and action
  • AnatomyModal shell, search, grouped results.
  • UsageQuick actions and route jumping.
  • AccessibilityLabelled dialog semantics required.
  • CodeMax width 720px.

22. Tabs

Parallel content sections
  • AnatomyTab list, tab, indicator, panel.
  • UsageSame-level content partitions.
  • AccessibilityArrow keys move between tabs.
  • CodeKeep labels concise and stable.

23. Accordion

Progressive disclosure
  • AnatomyHeader, icon, panel body.
  • UsageDense secondary content and filters.
  • AccessibilityExpose expanded state.
  • CodeAvoid janky height transitions.

24. Breadcrumb

Hierarchical orientation
  • AnatomyList, separators, current item.
  • UsageDeep content or app hierarchies.
  • AccessibilityUse breadcrumb navigation landmark.
  • CodeCollapse middle items on narrow widths.

25. Pagination

Large result sets
  • AnatomyPrev, page items, next, optional size control.
  • UsageWhen infinite scroll is not suitable.
  • AccessibilityCurrent page and disabled state announced.
  • Code40px minimum controls.

26. Card

Summary, preview, or grouped content
  • AnatomyContainer, optional media, title, body, actions.
  • UsageCollections, summaries, previews.
  • AccessibilityWhole-card click only if unambiguous.
  • Code24px padding, elevation shift on hover.

27. List item

Navigation and dense rows
  • AnatomyLeading visual, content block, trailing action/meta.
  • UsageSettings, indexes, navigation lists.
  • AccessibilityConsistent hit area across row.
  • Code56px comfortable height, 44px compact.

28. Table

Structured, scannable data
  • AnatomyHeader, rows, cells, sort affordance.
  • UsageData-rich operational content.
  • AccessibilityMaintain real table semantics.
  • Code44/52px rows, tabular numerals.

29. Data visualization card

Metric plus interpretation
  • AnatomyTitle, metric, chart, legend, controls.
  • UsageTrends and comparisons with immediate takeaway.
  • AccessibilityInclude textual summary or table backup.
  • CodeUse neutral + primary + one support tone.

30. Modal dialog

Focused blocking decision
  • AnatomyOverlay, panel, title, body, footer, close.
  • UsageCritical decisions and focused tasks.
  • AccessibilityTrap focus, support escape when appropriate.
  • Code560px standard width.

31. Drawer / sheet

Secondary workflow surface
  • AnatomyOverlay, panel, header, body, footer.
  • UsageFilters, details, side tasks.
  • AccessibilityFollow modal rules when blocking.
  • CodeRight drawer desktop, bottom sheet mobile.

32. Toast

Lightweight transient feedback
  • AnatomyStatus icon, message, action, dismiss.
  • UsageNon-blocking confirmations.
  • AccessibilityUse appropriate live region politeness.
  • CodeAuto-dismiss only for neutral/success.

33. Alert banner

Persistent contextual message
  • AnatomyContainer, icon, message, optional action.
  • UsageWarnings, information, feature state.
  • AccessibilityDo not steal focus unless critical.
  • CodeCan appear full-width or inline.

34. Empty state

First-use or no-result guidance
  • AnatomyVisual, title, description, actions.
  • UsageGuide next step clearly.
  • AccessibilityActions must be explicit and descriptive.
  • CodeKeep within 480px content width.

35. Skeleton loader

Layout-matched loading state
  • AnatomyShimmer blocks mirroring eventual layout.
  • UsageContent-heavy loads over spinners.
  • AccessibilityMark region busy, not individual blocks.
  • CodeSubtle shimmer only.

36. Progress indicator

Task progression or system work
  • AnatomyTrack, fill, optional label/value.
  • UsageUploads, setup steps, generation.
  • AccessibilityExpose min, max, and current values.
  • Code4px default, 8px emphasized.

37. Avatar

People and ownership markers
  • AnatomyImage or initials, frame, status dot optional.
  • UsageProfiles, collaborators, ownership.
  • AccessibilityMeaningful alt text where relevant.
  • Code24, 32, 40, 56px sizes.

38. Navigation bar

Top-level route structure
  • AnatomyBrand, nav items, utilities, profile area.
  • UsageProduct and marketing shells.
  • AccessibilityCurrent route must be programmatically clear.
  • Code64px desktop, 56px mobile.

39. Sidebar navigation

Dense internal wayfinding
  • AnatomySection labels, items, icons, active rail.
  • UsageSettings and multi-area products.
  • AccessibilityCollapsed labels still need accessible names.
  • Code280px expanded, 80px collapsed.

40. Footer

Utility closure and trust
  • AnatomyUtility links, legal, support, optional status.
  • UsageSite-wide support and completion.
  • AccessibilityOrganize links semantically.
  • CodeLow visual weight with strong top divider.
Patterns

How components work together

Patterns prevent teams from inventing one-off assemblies. They define repeatable ways to structure forms, search, navigation, feedback, and mobile behavior.

Forms

Prefer single-column forms, group related inputs, validate with restraint, and keep helper text beneath fields rather than inside placeholders.

Search and filters

Search leads, filters support. Keep common filters visible and move advanced controls into a drawer or sheet.

Navigation

Use top nav for broad product areas, side nav for internal architecture, and breadcrumbs only where hierarchy truly matters.

Feedback

Use inline validation for field issues, toast for lightweight confirmation, banners for persistent context, and modal confirmations for irreversible actions.

Developer guide

Design-to-code implementation model

The system should compile from token source to semantic theme layer to component API. Product teams compose from the shared vocabulary instead of inventing visual rules locally.

Implementation guidance

  • Keep primitive and semantic tokens separate.
  • Drive themes with root-level data attributes.
  • Normalize component props around size, variant, tone, disabled, and loading.
  • Prefer enums over many booleans in component APIs.

Engineering QA

  • Test light, dark, reduced motion, and 200% zoom.
  • Review empty, loading, error, disabled, and success states.
  • Check long labels and localization expansion.
  • Validate keyboard and screen reader behavior in Storybook or an equivalent harness.
Governance

How the system stays coherent

Good systems degrade when exceptions feel cheaper than alignment. Governance keeps the language coherent without slowing teams down.

Do

  • Require anatomy, states, usage, accessibility, and code notes for every new component.
  • Version tokens and document migration paths.
  • Review changes to foundations against accessibility and regression criteria.
  • Compose patterns from shared parts before proposing new primitives.

Don’t

  • Introduce one-off shadows, spacing, or radius values inside product teams.
  • Use color or motion alone to carry meaning.
  • Ship components without disabled, focus, and error states.
  • Allow marketing flourish to reshape core product UI behavior.
